## Decisões técnicas

1. **Implementação direta em `main`, sem branch/worktree**: consistente com o histórico do repo
   (nenhum commit anterior usou branch/PR). Confirmado explicitamente com o usuário antes de
   começar, por ser regra dura da skill `subagent-driven-development` (nunca implementar em
   main/master sem consentimento explícito).
2. **Modelo `haiku` pros implementadores, `sonnet` pros revisores por task, `opus` pra revisão
   final**: o plano continha código completo em cada task (transcrição+teste, não invenção), então
   o modelo mais barato bastava pra implementação; revisão pede mais julgamento (contraste WCAG,
   armadilhas de classe Tailwind composta dinamicamente), por isso um tier acima; a revisão final
   de branch cobre os 4 commits juntos, por isso o modelo mais capaz.
3. **Fix de 1 linha aplicado direto pelo controller, sem subagente**: a skill orienta usar
   subagente pra achados de revisão, mas pra uma mudança de uma linha (adicionar um
   `assert.doesNotMatch`) o overhead de um dispatch completo não se justificava — decisão de
   pragmatismo, não um desvio da skill em espírito (a mudança foi verificada rodando a suíte
   completa antes de commitar).

## Sessão — 21:30 (continuação, via `dinya-app`) — Cadastro/login de cliente + carrinho sincronizado

### Contexto

Trabalho conduzido a partir de uma sessão do `dinya-app` (repo irmão) que também tocou este repo:
usuário pediu pra puxar a pendência antiga da "loja virtual" — o carrinho aqui
(`lib/carrinho.tsx`, ver `memory/decisions/2026-07-21-ajustes-ui-linha-preco-carrinho.md`) era só
`localStorage`, sem conta. Pedido: cadastro, login, carrinho sincronizado entre dispositivos.

### Entregáveis

- `lib/api-cliente.ts` (fetch pro `dinya-app`) + `lib/auth-cliente.tsx` (contexto de sessão,
  token em `localStorage`) — novo `NEXT_PUBLIC_DINYA_API_URL` (client-side; diferente de
  `DINYA_API_URL`, que só existe em build-time no `sync-catalogo.mjs`), fallback pra
  `https://api.dinya.com.br` no código.
- Páginas `/conta`, `/conta/login`, `/conta/cadastro` (com seção opcional de endereço). Navbar
  ganha ícone de conta (`FaRegUser`).
- `lib/carrinho.tsx`: `ItemCarrinho` ganha `codigoErp`; convidado continua funcionando igual
  (decisão confirmada com o usuário do lado do `dinya-app`); ao logar, mescla o carrinho local
  com o da conta (`POST .../carrinho/mesclar`) e passa a espelhar add/remove no backend.
- **Achado real ao testar em Chromium de verdade (Playwright)**: a conversão da resposta do
  backend pra `ItemCarrinho` usava `slug: i.codigo` (código do ERP) — quebrava silenciosamente
  o "já adicionado"/remover, já que o resto do site sempre usa `slugify(nome)` como slug.
  Corrigido pra `slug: slugify(nome)`. `npm test`/`npm run build` sozinhos não pegariam isso
  (não executam JS no navegador) — só apareceu rodando de ponta a ponta com Playwright real.

### Verificação
